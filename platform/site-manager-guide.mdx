This guide is for librarians, archivists, publishers, collection managers, curators, and other institutional staff who are responsible for site operations but don't need day-to-day terminal access.

<Info>
**What you can do from the Dashboard:** launch new sites, request changes through the LibOps Task Agent, manage team access, lock down access with firewall rules, and coordinate your production launch. No terminal. No deep technical expertise required.
</Info>

## Prototyping ideas with the LibOps Task Agent

Start from plain text in the Dashboard or Slack. The Task Agent does the technical work; you decide whether the result is acceptable.

<Steps>
  <Step title="From plain text to production" icon="message">
    Write the request — a new homepage banner, a plugin configuration, a UI adjustment — in the Dashboard or Slack.
  </Step>
  <Step title="Validate the preview" icon="eye">
    LibOps implements the change and provisions a live **preview environment**. Test and validate exactly what your users will see.
  </Step>
  <Step title="Technical review" icon="code-branch">
    Behind the scenes, the agent opens a GitHub pull request. Your technical staff (or LibOps reviewers) confirm the code is secure and follows best practices.
  </Step>
  <Step title="Launch to production" icon="rocket">
    Once you've accepted the preview and the technical review is complete, the change deploys to production.
  </Step>
</Steps>

## Provisioning a new open-source site

Most site management workflows begin in the Dashboard.

<Steps>
  <Step title="Pick the organization and project" icon="folder-tree">
    Identify the institutional organization and project that should own the new site.
  </Step>
  <Step title="Create the site" icon="circle-plus">
    Select **Create site** and choose one of the supported application templates.
  </Step>
  <Step title="Configure the repository" icon="github">
    Decide whether LibOps should manage the site's repository or connect to your own GitHub organization.
  </Step>
  <Step title="Set up the environment" icon="gear">
    Name the site, choose its purpose (production, staging, etc.), and set the intended hostname.
  </Step>
  <Step title="Review and launch" icon="check">
    Confirm the generated configuration — repository, access, hostnames — and finalize the site.
  </Step>
</Steps>

<Warning>
LibOps only supports workloads based on the documented templates. Running other applications on managed infrastructure is not permitted. See the [Terms of Service](https://www.libops.io/terms/) for details.
</Warning>

## Working with the LibOps Task Agent

Use the Task Agent for site enhancements you can validate in a preview environment.

<CardGroup cols={2}>
  <Card title="UI / UX refinement" icon="paintbrush">
    Adjust homepage banners, navigation, theme tweaks, and presentation polish.
  </Card>
  <Card title="Application configuration" icon="sliders">
    Set up new plugins, modules, or theme-specific settings.
  </Card>
  <Card title="Feature exploration" icon="lightbulb">
    Modify content models or workflow behaviors to try a new approach.
  </Card>
  <Card title="Troubleshooting" icon="bug">
    Investigate errors or behavior visible in the preview environment.
  </Card>
</CardGroup>

<Check>
The Task Agent **never** modifies production directly. It always works on a task branch, deploys to a preview environment, and waits for your sign-off — and your engineers' code review — before anything reaches production.
</Check>

## Slack integration

When enabled, the LibOps Slack integration lets you start development tasks without leaving your primary communication channel — great for concise requests where the site and desired outcome are clear.

```text Slack
/libops chat [org_id] project [project_id] site [site_id] Add a homepage alert banner that can be toggled by the editorial team.
```

<Tip>
LibOps posts status updates and a link to the preview environment back in the Slack thread once the implementation is ready for your review.
</Tip>

For setup and the full command reference, see the [Slack Integration](/platform/slack) page.

## Managing site privacy and launch

By default, every LibOps site is protected by an IP allowlist. Development, staging, and preview environments are only reachable by authorized staff and reviewers until you decide to publish.

When you're ready to take a production site public, use the Dashboard to:

<Steps>
  <Step title="Verify production state" icon="clipboard-check">
    Confirm the production environment reflects all approved changes.
  </Step>
  <Step title="Configure the domain" icon="globe">
    Add the custom domain and verify the application's hostname settings.
  </Step>
  <Step title="Open public access" icon="door-open">
    Adjust the firewall rules to allow public traffic — only once the site is approved for launch.
  </Step>
</Steps>

For a complete pre-launch checklist, see [Getting a Site Ready to Publish](/platform/getting-site-ready-to-publish).

## Collaborating with technical teams

LibOps is designed to reduce the need for deep infrastructure expertise — but some changes still benefit from technical oversight. Bring in your engineers or LibOps support when a task involves:

<CardGroup cols={3}>
  <Card title="Security & compliance" icon="shield-halved">
    Changes affecting authentication, privacy, or data export.
  </Card>
  <Card title="Core architecture" icon="diagram-project">
    Significant data model changes or large-scale migrations.
  </Card>
  <Card title="Third-party integrations" icon="plug">
    Services that require privileged access into your stack.
  </Card>
</CardGroup>

The platform's collaborative review model is the same in both cases: you validate the functional behavior in the preview environment; technical staff inspect the underlying implementation in GitHub.
