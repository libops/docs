LibOps gives teams three operating surfaces: the dashboard, the full platform API, and the `sitectl` command-line utility.

That matters because institutions rarely have a single type of operator. Some work belongs in a UI, some belongs in automation, and some belongs in a terminal next to the application repository.

## Full Platform API

The LibOps API exposes the core platform resources:

- Organizations.
- Projects.
- Sites.
- Settings.
- Secrets.
- Firewall rules.
- Members.
- SSH keys.
- API keys.
- Deployment and status operations.

Teams can use the API to integrate LibOps with provisioning systems, reporting workflows, governance tooling, or internal service catalogs.

## sitectl

`sitectl` is the LibOps command-line utility for operating local and remote Docker Compose sites.

It is built for teams that run production open-source software, often across multiple applications, without making every project depend on a dedicated DevOps hire.

The more institutional knowledge that lives in `sitectl`, the fewer institutions get blocked from adopting the software they need.

`sitectl` uses Docker Compose as its operational primitive so that:

- **Institutions** can reliably host the software they depend on without staffing a dedicated DevOps team.
- **Individual contributors** get standardized tooling that removes environmental toil, so they can focus on the application work that matters.

## Turn Best Practices Into Code

`sitectl` helps operators run Compose projects in production by turning common practices into repeatable commands.

The more institutional knowledge that lives in the tool, the less it depends on any one person.

## Why teams use both

Common tooling lowers the cost of adopting open-source software. When more GLAM institutions can run, understand, and maintain their own instances, the surrounding software communities get more capable participants.

LibOps uses the API and `sitectl` together so platform operations can support both ends of that spectrum: centralized automation for repeatability, and hands-on command-line workflows for people doing the application work.
