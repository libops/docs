LibOps uses GitHub repositories as the source of truth for site code, deployment configuration, task branches, and pull requests. A site can use a LibOps-managed repository by default, or it can use a repository in your own GitHub organization after you install the LibOps GitHub App.

## Repository Options

### LibOps Managed

LibOps Managed is the default option when creating a site. It does not require the customer to create a GitHub account or install anything in their own GitHub organization.

When you choose LibOps Managed, LibOps copies the selected template into a private repository in a LibOps-managed GitHub organization. The site is connected to that repository, and LibOps uses it for deployments, preview environments, task branches, and pull requests.

Use LibOps Managed when:

- You want LibOps to own the repository operations for the site.
- Your team does not already have a GitHub organization for this site.
- You want the fastest path from template selection to a running site.

### Customer GitHub Organization

Use a customer-owned GitHub organization when your team wants to keep the site repository under its own GitHub org, use its own review process, or connect existing development workflows.

To connect a customer-owned repository:

1. A GitHub organization owner or app manager opens the LibOps GitHub App installation link from the LibOps dashboard.
2. Select the GitHub organization that should own the site repository.
3. Grant the LibOps app access to the selected site repository, or to all repositories if your organization prefers that model.
4. Return to LibOps and connect the GitHub organization. LibOps stores the GitHub App installation ID for that GitHub repository owner.
5. Create the site with the customer repository URL, or choose a supported template and have LibOps create the repository in your organization when that workflow is available.

### GitHub App Permissions

GitHub shows the requested repository access and permissions during app installation. The exact install screen can change as LibOps features evolve, but these are the permissions you should expect to approve for the LibOps GitHub App:

| GitHub permission | Access | Why LibOps uses it |
| --- | --- | --- |
| Repository access | Selected repositories or all repositories | Selected repositories is enough when LibOps works only with existing site repositories you choose. All repositories, or the equivalent repository-creation access shown by GitHub, is needed when LibOps should create new site repositories in your organization over time. |
| Metadata | Read | Required by GitHub Apps and used by LibOps to identify repositories, owners, and installations. |
| Contents | Read and write | Clones and pulls site code for deployments, writes **LibOps Task Agent** commits to `task/*` branches, and copies approved templates into new site repositories when repository creation is enabled. |
| Pull requests | Read and write | Opens and updates **LibOps Task Agent** pull requests, reads pull request branch context, and supports PR review feedback loops. |
| Issues | Read and write | GitHub exposes pull request conversation comments through the issues API. LibOps uses this to post addressed-in-commit acknowledgements back to pull requests after reviewer feedback is handled. |
| Checks | Read and write | Reports LibOps automation status on GitHub commits when the platform handles repository push events. |
| Deployments | Read and write | Creates and updates GitHub deployment statuses for site deploys started from repository events. |
| Administration or repository collaborators | Write, only when enabled | Creates or manages LibOps site repositories and syncs LibOps organization, project, and site membership to GitHub repository collaborators. |

If the app is limited to selected repositories, LibOps can work only with those repositories. Add every site repository that should use LibOps deployments, **LibOps Task Agent** tasks, pull requests, preview environments, or PR review follow-up. If you want LibOps to create new site repositories inside your GitHub organization, the installation must allow repository creation for that organization.

Permissions marked as conditional are only needed for installations that use those features. For example, repository collaborator management is required when LibOps member-to-repository sync is enabled, but it is not part of the normal task-branch token used by a **LibOps Task Agent**.

One LibOps organization can connect repositories from more than one GitHub organization. LibOps stores each GitHub App installation ID by GitHub repository owner and chooses the installation that owns the repository for task branches, pull requests, collaborator sync, and automated webhook follow-up.

## Task Branches, Preview Environments, and Pull Requests

Coding tasks run against a development or ephemeral preview environment, not against production directly. When a coding task starts, LibOps creates or selects a live preview environment for the site and sends the task to the **LibOps Task Agent** with access to that environment's repository checkout.

The **LibOps Task Agent** works on a task branch and opens or reuses a GitHub pull request. The preview environment gives internal stakeholders a running site where they can review the change, test workflows, validate content, and complete quality assurance before production changes move forward.

The task remains open until the pull request is merged or closed. Production changes should only move forward after the pull request is reviewed by a subject matter expert for that project and the preview environment has been accepted by the people responsible for the site.

If your organization does not have someone available to review a coding task, contact LibOps about review credits. LibOps can help inspect AI-generated code, respond to review needs, and support merging approved changes.

## Review Comment Feedback Loop

GitHub pull request review comments can feed back into the original LibOps task when the reviewer explicitly invokes LibOps. When reviewers comment on a PR created by the **LibOps Task Agent** and include `@libops` in the comment body, LibOps can listen for those GitHub review events, route the comments back to the originating task, and let the agent respond with follow-up commits.

Reviewer comments are only eligible for agent follow-up when the comment includes `@libops` and the GitHub reviewer has write-or-better permission on the pull request repository. LibOps member management syncs repository collaborators for accounts that have a GitHub username: owner/developer membership at the organization, project, or site level grants write access to affected site repositories, and viewer membership grants read access. This keeps normal GitHub review ergonomics while using explicit invocation and GitHub repository permission as the webhook-time authorization signals.

This makes a PR review comment behave like a follow-up chat message, while keeping the review process in GitHub:

- A reviewer comments on a line, file, or pull request conversation and mentions `@libops`.
- LibOps maps the event to the **LibOps Task Agent** task and branch.
- The agent reviews the comment and updates the task branch where appropriate.
- The preview environment can refresh so stakeholders can verify the change.
- The pull request remains the approval point before production rollout.

Example review loop:

1. A Slack request starts a coding task for a site and LibOps opens pull request `https://github.com/example/site/pull/42` from branch `task/12345678-1234-1234-1234-123456789abc`.
2. A reviewer comments, "`@libops` please add a regression test for the empty search state."
3. GitHub sends the review event to LibOps. LibOps verifies the `@libops` invocation, that the reviewer has write-or-better repository permission, and that the pull request maps to the original task branch.
4. The comment is appended to the task as follow-up input.
5. The **LibOps Task Agent** updates the same branch and pushes another commit to the same pull request.
6. The platform posts a pull request comment that links to the commit that addressed the feedback.
7. The completion surfaces show the same result fields:

```text
LibOps task `12345678` is ready.
Preview: https://preview.example.edu
Pull request: https://github.com/example/site/pull/42
Summary:
Added the empty-state regression test and adjusted the search template fallback.
```

## Repository Member Sync

When a LibOps account has a GitHub username, LibOps membership changes update the relevant GitHub repository collaborators. Organization membership applies to the organization's site repositories, project membership applies to that project's site repositories, and site membership applies to that site's repository. If multiple sites point at the same repository, LibOps keeps the highest effective permission for that user.

Manual GitHub collaborator changes can diverge from LibOps membership and may be overwritten by the next LibOps membership update. For customer-owned repositories, the LibOps GitHub App installation must include the repository permissions required to manage collaborators.

## Token Handling

The LibOps GitHub App private key is stored in the LibOps API Vault. Site VMs and **LibOps Task Agent** tasks never receive the app private key.

When a task needs GitHub access, the LibOps API mints a short-lived GitHub App installation token for the relevant installation. The request is validated against the site repository and the allowed `task/*` branch before a token is returned. Tokens are used at runtime by Git helpers and are not written into repository config.

For LibOps Managed sites, the token comes from the LibOps app installation on the platform's managed GitHub organization. For customer-owned repositories, the token comes from the app installation for the GitHub owner that owns the repository. LibOps does not use personal access tokens for repository mutation.
