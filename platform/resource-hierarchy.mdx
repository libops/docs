LibOps resources are organized in three levels: organizations, projects, and sites. Members, secrets, settings, and firewall rules attach at one of those levels depending on how broadly they should apply.

<CardGroup cols={3}>
  <Card title="Organizations" icon="building-columns">
    The institution, department, program, agency, or customer account that owns work in LibOps.
  </Card>
  <Card title="Projects" icon="folder-tree">
    A group of related sites, usually one application or service family inside an organization.
  </Card>
  <Card title="Sites" icon="globe">
    A specific running environment such as production, staging, or a review environment.
  </Card>
</CardGroup>

## How Scope Works

Organization-level resources apply across the organization. A member attached to an organization can work across that organization's projects and sites according to their role.

Project-level resources apply to one project and all sites under it. Use this when a person or configuration should apply to one application but not the whole organization.

Site-level resources apply to exactly one site. Use this for production-only access, one-off firewall entries, or settings that should not affect staging or other environments.

<Info>
When access should be limited, attach it at the narrowest level that matches the work.
</Info>

## Environments And Runtime Hosts

A site record represents one running environment for a codebase: production, staging, review, development, or another named environment. A project can have more than one runtime host. Production sites use production-only runtimes, while non-production sites use separate ephemeral or custom runtimes. Default and role-based placement assigns no more than seven compatible sites to one runtime and creates a slot-stable overflow runtime when the current one is full; explicitly selecting a full runtime returns a capacity error.

Terraform and cloud-compose create the runtime host, install the base operating-system dependencies, and place the initial Compose project scaffolding on the VM. They do not manage day-to-day state inside the VM. The site reconciler running on the VM handles operational state such as SSH users, allowed keys, secrets files, firewall rules, and GitHub-triggered deployments.

That split affects how scopes are applied:

- Organization and project resources are inherited by every environment under that scope.
- Site resources are environment-specific overrides. The happy path is to keep shared behavior at project scope, let non-production environments inherit the same settings, and override only the production site when production needs to differ.
- Site-level members for production apply only to the production site.
- Site-level members for non-production environments in the same project are treated as the non-production access set for that project.

## Members

Members can be attached to organizations, projects, or sites:

| Attachment | Applies To | Typical Use |
| --- | --- | --- |
| Organization | All projects and sites in the organization | Internal platform owners, agency account leads, broad technical staff |
| Project | Every site in one project | Application maintainers for one service |
| Site | One environment | Production approvers, temporary support, limited environment access |

Member roles are `owner`, `developer`, and `read`. LibOps creates or resolves member accounts by email when adding a member. Existing members are still identified by account ID for update and delete operations.

LibOps administrators can add any email address. Other organization administrators follow the same approved email-domain policy used during signup.

## Agencies And Related Organizations

Organizations can have approved relationships with other organizations. This supports agencies, vendors, consortia, and central service teams that manage work for customer or partner organizations.

In that model, the agency keeps its own LibOps organization and members. After the relationship is approved, agency members can be granted access to the customer organization's resources without adding every agency staff account directly to the customer organization.

This keeps ownership clear:

- The customer organization remains the owner of its projects and sites.
- The agency organization manages its own staff membership.
- The relationship defines which organization can operate on behalf of the other.

## Other Scoped Resources

The same hierarchy applies to common operational resources:

| Resource | Organization Scope | Project Scope | Site Scope |
| --- | --- | --- | --- |
| Secrets | Shared by all projects and sites | Shared by all sites in a project | Available only to one site |
| Settings | Organization-wide defaults | Project defaults | Site-specific overrides |
| Firewall rules | Broad allow or block policy | Application-level rules | One environment only |
| GitHub access | Repositories for the organization | Repositories for one project | One site repository |

Use organization scope for policy and shared defaults. Use project scope for application-level operations. Use site scope for environment-specific changes.
