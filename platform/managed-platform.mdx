LibOps gives institutions a managed platform for operating open-source applications — without the need to build a custom cloud operations practice for every project.

The platform combines Google Cloud infrastructure, single-tenant virtual machines, Cloudflare edge security, Vault-backed secret management, GitHub-centric deployment workflows, and standardized Docker Compose patterns into one operating model.

<Info>
**Who this page is for:** institutional leaders, IT directors, and procurement evaluators. For day-to-day operators see the [Site Manager Guide](/platform/site-manager-guide). For developers see the [Engineer Guide](/platform/engineer-guide).
</Info>

## Single-tenant site isolation

Every LibOps site runs on its own dedicated virtual machine. Each production workload has a distinct compute boundary, avoiding the risks of shared application runtimes.

<CardGroup cols={2}>
  <Card title="Fault isolation" icon="shield">
    App-specific issues stay contained within the site's own environment.
  </Card>
  <Card title="Predictable performance" icon="gauge">
    Resource allocation and maintenance windows are scoped to the individual site.
  </Card>
  <Card title="Clear ownership" icon="user-check">
    Operational responsibility for production, staging, and review environments is explicitly defined.
  </Card>
  <Card title="Simpler troubleshooting" icon="magnifying-glass">
    Easier to reason about when diagnosing performance or configuration issues.
  </Card>
</CardGroup>

## Standardized application lifecycle

Every LibOps template follows the same operational shape. Site managers work through the Dashboard; technical staff use the API and `sitectl`; the underlying repository and Docker Compose workload stay consistent across environments.

<Steps>
  <Step title="Create the site" icon="folder-plus">
    Start from a supported template for the application family.
  </Step>
  <Step title="Configure access" icon="shield">
    Set secrets, members, firewall rules, and domains through platform resources.
  </Step>
  <Step title="Preview changes" icon="eye">
    Review branch and task work in live preview environments.
  </Step>
  <Step title="Review in GitHub" icon="github">
    Keep code, configuration, and deployment logic in pull requests.
  </Step>
  <Step title="Roll out with sitectl or the Dashboard" icon="rocket">
    Promote approved changes through the same production path.
  </Step>
</Steps>

<Tip>
Technical teams still get the same repo-level Make targets across templates. See the [Engineer Guide](/platform/engineer-guide) and [Application Templates](/templates/app-support) for the developer-facing details.
</Tip>
