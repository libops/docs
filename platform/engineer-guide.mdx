This guide is for software engineers, systems administrators, developers, and technical partners who build and operate applications on the LibOps platform.

<Info>
LibOps is a transparent platform, not a black box. You get a full API, the `sitectl` CLI, GitHub-centric workflows, and standardized Docker Compose patterns — so you keep complete visibility and control over every stack you operate here.
</Info>

## One workload, laptop to production

Every LibOps-supported application follows a single operational pattern. The Docker Compose file in your repository is the **exact same workload** we run in production.

<Tip>
A consistent workload means: no environmental toil, no "works on my machine" debugging, and no need to learn a separate deployment system for each application family. An engineer can move between OJS, Islandora, Drupal, and Omeka using the same tooling.
</Tip>

## Two ways changes ship

LibOps gives engineers two equally first-class paths for getting work into production. Both end with the same review surface — a GitHub pull request and a preview environment.

<Tabs>
  <Tab title="Via the Task Agent" icon="wand-magic-sparkles">
    For changes that start as a Dashboard or Slack request.

    <Steps>
      <Step title="Task assigned" icon="message">
        A site manager (or you) defines the task on a specific site.
      </Step>
      <Step title="Branch implementation" icon="code-branch">
        The Task Agent implements the change on a dedicated `task/*` branch.
      </Step>
      <Step title="Pull request opens" icon="github">
        LibOps opens or updates a GitHub pull request for the change.
      </Step>
      <Step title="Preview environment" icon="globe">
        The change deploys to an ephemeral preview environment automatically.
      </Step>
      <Step title="Engineering review" icon="magnifying-glass-chart">
        You inspect the diff in GitHub against security and architectural standards.
      </Step>
      <Step title="Validation and merge" icon="rocket">
        Once you approve and stakeholders validate the preview, the change merges and rolls out.
      </Step>
    </Steps>

    <Tip>
    Mention `@libops` in a GitHub PR comment and the agent will iterate based on your feedback — pushing new commits to the same branch.
    </Tip>
  </Tab>

  <Tab title="Working locally" icon="laptop-code">
    For changes engineers prefer to write directly with their own tools.

    <Steps>
      <Step title="Clone and branch" icon="github">
        Standard `git clone`, `git checkout -b`. Your existing workflow.
      </Step>
      <Step title="Run the stack locally" icon="docker">
        `make build && make up` — same Compose project that runs in production.
      </Step>
      <Step title="Edit, test, push" icon="code">
        Make the change. `make test` runs the same suite as CI.
      </Step>
      <Step title="Open a PR" icon="code-pull-request">
        LibOps automatically provisions a preview environment for your branch.
      </Step>
      <Step title="Merge to roll out" icon="rocket">
        Standard rollout pipeline from there — no special hand-off.
      </Step>
    </Steps>

    <Tip>
    `sitectl` connects your local Compose context to remote sites — handy for pulling logs, managing secrets, or port-forwarding into production for debugging.
    </Tip>
  </Tab>
</Tabs>
